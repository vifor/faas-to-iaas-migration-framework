EXECUTIVE SUMMARY
-----------------
The PetStore FaaS application uses a minimal DynamoDB-based data layer consisting 
of TWO tables: petstoreFranchise and petstoreTenants. Pet and Order entities are 
NOT stored in DynamoDB; they exist only as authorization entities in the Amazon 
Verified Permissions (AVP) policy schema.

This architecture presents a significant gap between the FaaS implementation and 
the target IaaS monolith, which implements a complete relational data model 
including Pet and Order tables.

3. User / Group
   -------------
   AVP Entity Types: MyApplication::User, MyApplication::Group
   
   IMPLEMENTATION:
   User and Group entities are NOT stored in DynamoDB. They are derived from 
   Cognito JWT token claims at runtime:
   - User identity: cognito:username
   - Group memberships: cognito:groups
   - Franchise affiliations: custom:franchiseCode (comma-separated)
   - Store employment: custom:employmentStoreCode (comma-separated)
   
   IMPACT:
   User management is delegated to Amazon Cognito. The IaaS migration must 
   either replicate this Cognito integration or implement a new user/role 
   management system.

ENFORCED (via AVP Schema):

NOT ENFORCED (DynamoDB has no foreign key constraints):
--------------------------------------------------------
- Franchise.stores array is not validated (may contain non-existent store IDs)
- Store.franchise.id is not validated (may reference non-existent franchise)
- No cascade delete behavior (deleting a franchise does not remove its stores)

MIGRATION COMPLEXITY FACTORS
================================================================================

1. INCOMPLETE DATA MODEL
   - The FaaS application is a partial implementation (admin CRUD only)
   - Pet and Order business logic is NOT implemented
   - The IaaS target monolith has a complete data model with Pet, Order, and 
     additional tables

2. KEY SCHEMA MISMATCH
   - DynamoDB: petstoreTenants uses composite key (id + value)
   - IaaS Monolith: Store table likely uses a single primary key
   - The "value" sort key appears to be a design artifact with no clear purpose

3. NO DATA TO MIGRATE
   - Since Pet and Order tables do not exist in the FaaS implementation, there 
     is NO DATA to migrate for these entities
   - Only Franchise and Store data would be migrated

4. AUTHORIZATION ARCHITECTURE CHANGE
   - The FaaS application heavily relies on AVP (Amazon Verified Permissions) 
     with Cedar policy language
   - The IaaS monolith must implement equivalent authorization logic, likely 
     using application-level RBAC or a different authorization framework

5. COGNITO INTEGRATION
   - The FaaS application uses Cognito for authentication and user attributes
   - The IaaS migration must either maintain Cognito integration or implement 
     a new authentication system

================================================================================

For Migration to IaaS Relational Database:

1. NORMALIZE KEY SCHEMA
   - Store: Use a single primary key (store_id) instead of composite key
   - Remove the "value" attribute unless business logic requires it

2. IMPLEMENT FOREIGN KEY CONSTRAINTS
   - Store.franchise_id → Franchise.id
   - Pet.store_id → Store.id (new table)
   - Order.store_id → Store.id (new table)
   - Order.pet_id → Pet.id (new table)

3. CREATE MISSING TABLES
   - Pet table (as per OpenAPI spec + IaaS monolith schema)
   - Order table (as per OpenAPI spec + IaaS monolith schema)
   - Optionally: User table (if moving away from Cognito)

4. ADD INDEXES
   - Store.franchise_id (for join queries)
   - Pet.store_id (for store inventory queries)
   - Pet.status (for availability filtering)
   - Order.store_id (for store order history)
   - Order.customer_id (for user order history)
   - Order.status (for order processing workflows)

5. DATA TYPE CONVERSIONS
   - Franchise.stores: Convert from JSON array to relational 1:N relationship 
     (queried via Store.franchise_id)
   - Store.franchise: Convert from nested object to foreign key column

6. AUDIT FIELDS
   - Add created_at, updated_at timestamps to all tables (not present in FaaS)
   - Add soft delete flags if required (deleted_at)

MIGRATION SCOPE RECOMMENDATION
================================================================================

Based on the database discovery analysis, this section provides a documented 
recommendation on migration scope with technical justification.

RECOMMENDATION: TWO-PHASE MIGRATION (Backend First, Data Migration Deferred)

Rationale:
----------

The analysis reveals a fundamental architecture mismatch between the FaaS source 
and IaaS target that makes a single-phase migration impractical:

1. INCOMPLETE SOURCE IMPLEMENTATION
   -------------------------------------
   FaaS Status:
   - 2 DynamoDB tables implemented (petstoreFranchise, petstoreTenants)
   - Pet and Order entities: Authorization schema only, NO data persistence
   - Store operations Lambda: Authorization wrapper, NO business logic
   - Total functionality: Admin CRUD for Franchise and Store metadata
   
   IaaS Target Status:
   - 4 complete domain entities (Franchise, Store, Pet, Order)
   - Full business logic for pet inventory and order management
   - Comprehensive data model with audit fields and relationships
   
   IMPACT: There is NO Pet or Order data to migrate from FaaS to IaaS because 
   these tables do not exist. A data migration is unnecessary and impossible 
   for 50% of the target schema.

2. TABLE COUNT & COMPLEXITY
   -------------------------
   - Only 2 source tables (very low complexity)
   - No Global Secondary Indexes to replicate
   - No complex nested documents or binary attributes
   - Total migration effort: LOW
   
   IMPLICATION: The data migration is simple enough to be handled as a separate, 
   low-risk phase AFTER the backend code is migrated and validated.

3. DATA DUPLICATION PATTERNS
   ---------------------------
   - Franchise.stores array duplicates Store.franchiseId references
   - This denormalization is a DynamoDB pattern that does NOT translate directly 
     to relational schemas
   
   MIGRATION STRATEGY:
   - Phase 1 (Backend Migration): Refactor code to use relational join queries 
     instead of denormalized arrays
   - Phase 2 (Data Migration): Migrate Franchise and Store data with transformation 
     logic to normalize relationships
   
   RISK REDUCTION: Decoupling backend and data migrations allows testing the 
   new relational query logic against synthetic test data before touching 
   production data.

4. MISSING BUSINESS LOGIC
   -----------------------
   The FaaS application is an MVP prototype that only implements administrative 
   metadata management. The IaaS target must implement NEW features:
   - Pet inventory CRUD operations
   - Order placement and fulfillment workflows
   - Store inventory queries
   - Customer-facing search and ordering APIs
   
   DEVELOPMENT APPROACH:
   - Migrate existing Franchise/Store admin CRUD (backend + data)
   - Develop NEW Pet and Order features from scratch in IaaS
   - No data migration needed for Pet/Order (no source data exists)
   
   This is fundamentally a NEW FEATURE DEVELOPMENT project, not a pure migration.

5. AUTHORIZATION ARCHITECTURE MIGRATION
   -------------------------------------
   The FaaS application's core complexity is in Amazon Verified Permissions (AVP) 
   with Cedar policy language. The IaaS target must implement equivalent 
   authorization, which is a BACKEND LOGIC concern, not a data concern.
   
   PRIORITY: Migrate and test the authorization logic first (Phase 1: Backend) 
   before introducing the complexity of production data migration.

PHASE 2: DATA MIGRATION (Deferred to Post-Validation)

Scope:
- Export Franchise data from DynamoDB (petstoreFranchise table)

- Transform data to match IaaS schema:

  * Remove Store.value sort key (or map to new schema field if business logic exists)
  * Add audit timestamps (createdAt, updatedAt) with reasonable defaults

- Validate referential integrity (all Store.franchiseId values reference existing Franchises)

Prerequisites:
- Phase 1 backend migration COMPLETE and VALIDATED
- IaaS application deployed to staging environment
- Data transformation scripts tested against production snapshots

Deliverables:
- Migrated Franchise and Store records in IaaS data store
- Data validation report (row counts, integrity checks, sample comparisons)
- Rollback plan (data export from IaaS for disaster recovery)

Validation Criteria:
- 100% of Franchise records migrated successfully
- 100% of Store records migrated with correct franchise associations
- No orphaned references (all Store.franchiseId → Franchise.id validated)
- IaaS API returns identical results for existing Franchise/Store queries

Timeline Estimate: 1-2 weeks (low complexity due to only 2 tables, no GSIs)

Risks:
- VERY LOW: Only 2 simple tables, no complex data types
- Data volume likely small (pet store franchise metadata, not transactional data)
- No downtime required if using phased cutover (run FaaS and IaaS in parallel, 
  route read traffic to IaaS, validate, then route write traffic)


PHASE 3: NEW FEATURE DEVELOPMENT (Pet & Order Management)
~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~
Scope:

- Implement Order entity CRUD operations (NEW feature, not migrated)

Note: This is OUT OF SCOPE for the FaaS-to-IaaS migration framework because

Timeline Estimate: 4-8 weeks (depends on business requirements complexity)


ALTERNATIVE REJECTED: SINGLE-PHASE MIGRATION
---------------------------------------------

A single-phase migration (backend + data together) was considered but REJECTED 
for the following reasons:

1. UNNECESSARY COUPLING
   - The 2 source tables are simple and can be migrated independently
   - Coupling backend and data migrations increases the blast radius of failures
   - If data migration fails, the entire release must be rolled back (including 
     working backend code)

2. TESTING COMPLEXITY
   - Single-phase requires production data in staging environment for validation
   - Two-phase allows backend validation with synthetic data first
   - Production data access may have security/compliance constraints

3. NO TIME SAVINGS
   - Data migration for 2 simple tables takes 1-2 weeks
   - This does NOT justify the risk of coupling it with backend migration
   - The team can work in parallel: backend migration → validation → data migration

4. ROLLBACK COMPLEXITY
   - Single-phase rollback requires reverting both code AND data
   - Two-phase allows rolling back code without touching data (or vice versa)

5. PET & ORDER ENTITIES
   - These entities have NO source data, so "data migration" is meaningless
   - Single-phase migration cannot address 50% of the target schema


RISK ASSESSMENT
----------------

TWO-PHASE MIGRATION RISKS:

LOW RISKS:
- Data consistency during migration (mitigated by read-only validation period)
- Schema evolution between phases (mitigated by version control and migration scripts)
- Timeline delays (Phase 2 is only 1-2 weeks, minimal impact)

MEDIUM RISKS:
- Changes to FaaS data during Phase 1 (new Franchises/Stores added in production)
  Mitigation: Use incremental sync or snapshot at cutover time
- Authorization logic differences between FaaS (AVP) and IaaS (custom RBAC)
  Mitigation: Comprehensive authorization test suite

HIGH RISKS:
- NONE identified


SINGLE-PHASE MIGRATION RISKS:

MEDIUM RISKS:
- Larger rollback scope (code + data)
- Longer validation period before cutover

HIGH RISKS:
- Production data corruption if data transformation logic has bugs
- Complete system outage if migration fails (cannot serve traffic from either FaaS or IaaS)
- Security/compliance violations if production data is copied to staging unsafely


FINAL RECOMMENDATION
--------------------

Proceed with TWO-PHASE MIGRATION:
1. Phase 1: Migrate backend code for Franchise and Store admin CRUD (2-4 weeks)
2. Phase 2: Migrate Franchise and Store data with transformation (1-2 weeks)
3. Phase 3: Develop Pet and Order features as new greenfield work (4-8 weeks, out of scope)

This approach:
✓ Minimizes risk by decoupling backend and data concerns
✓ Enables faster validation with synthetic data
✓ Provides clear rollback points at each phase boundary
✓ Acknowledges that 50% of the target schema has no source data to migrate
✓ Correctly frames Pet/Order development as new feature work, not migration
